Before Bank Selection

3 experiments
#C1 Benefit-first intro screen title 0.5sp ▼
Hypothesis
If we rewrite the intro title to lead with the user's outcome (action/benefit-first framing, implying no card is needed), then more users proceed to bank selection because the user already knows the merchant — the headline should answer "what will I be doing?" not restate context they already have.
✓ If it wins
Benefit-first framing at the entry point sets intent and reduces cold-start anxiety. Users who understand what they're about to do proceed more confidently.
✗ If it fails
The title isn't what causes hesitation at this step — users drop because of credential anxiety downstream, not because the intro headline is unclear.
#C2 Action-aligned intro CTA 0.5sp ▼
Hypothesis
If we change the intro CTA to describe what happens next ("Connect my bank") rather than generic progression ("Continue"), then click-through increases because the CTA sets the expectation of the next step and aligns with the action the user is about to take.
Variations to test
Connect my bank Pay with my bank Select my bank
✓ If it wins
CTAs that name the next action outperform generic progression labels. Naming the action reduces uncertainty about what "continue" means.
✗ If it fails
Users don't read CTA labels carefully at intro screens — they tap whatever button moves them forward regardless of the label.
#C3 Callout re-entry — measurement opportunity 0.5sp ▼
Hypothesis
If we instrument the "learn more" callout click and return rate, then we can determine whether users who click out return to complete the flow — informing whether this link is a trust-building asset or a dropout accelerator.
⚠️ Measurement annotation, not a copy change. Pair with any copy experiment on this screen to capture baseline behaviour.
✓ If it wins
Callout clicks have a meaningful return rate — the link is building trust. Optimise its copy and position.
✗ If it fails
Most users who click the callout do not return — it's a dropout accelerator. Remove or replace with inline content.

Bank Selection

5 experiments
#C4 Conversational screen title 0.5sp ▼
Hypothesis
If we change "Select your bank" to a more conversational tone, then more users engage with the bank list because a lower-friction framing reduces the transactional feel of the first real decision point.
Variations to test
Which bank do you use? Connect your bank Select your bank (control)
✓ If it wins
Conversational headers reduce perceived friction. Readability grade correlates with engagement on this screen.
✗ If it fails
Users scan directly to the bank list — copy tone at this level is too subtle to move the needle.
#C5 Search placeholder — natural language 0.5sp ▼
Hypothesis
If we change the search placeholder to more natural language, then search engagement increases because a friendlier, more explicit instruction reduces hesitation on first tap.
Variations to test
Search for your bank Search by bank name Find your bank (control)
✓ If it wins
Placeholder copy quality affects engagement. Explicit objects ("for your bank") reduce ambiguity for first-time users.
✗ If it fails
Users ignore placeholder text — they tap and type regardless of what it says.
#C6 "Most popular" label — geo-aware variation 0.5sp ▼
Hypothesis
If we replace "Most popular" with "Your region's top banks" when geo-location is available, then users feel the list is personalised and trust that their bank is more likely to appear.
Depends on: geo-location config
✓ If it wins
Personalised section labels increase perceived relevance. Even a copy-level personalisation signal lifts trust in the list.
✗ If it fails
"Most popular" is already understood as "relevant to me" — geo language raises privacy concerns or users don't notice it.
#C7 Returning user label — "Your bank" 0.5sp ▼
Hypothesis
If we label the returning user's previously used bank as "YOUR BANK" instead of "last used", then returning users feel a stronger sense of ownership and personal recognition, reducing time-to-selection.
Depends on: returning user detection
✓ If it wins
Ownership language ("YOUR BANK") outperforms behavioural history labels ("last used"). Personalisation at this micro level builds belonging.
✗ If it fails
"Last used" is clearer — "YOUR BANK" is ambiguous for users with multiple banks and may not accelerate selection.
#C8 No-results empty state — help-first framing 0.5sp ▼
Hypothesis
If we change the no-results state from "This bank isn't available" to help-first framing, then fewer users abandon because the copy shifts from a dead-end signal to an actionable prompt that keeps users in the flow.
Variations to test
Don't see your bank? Try a different spelling Can't find your bank? Check the spelling Your bank may not be listed yet
✓ If it wins
Help-first empty states retain users who would have read "isn't available" as permanent. Keeps them trying rather than accepting a dead end.
✗ If it fails
Users who see no results searched accurately — reframing doesn't change the underlying absence of their bank.

Native Bank Sign In

#C9 Subtitle — remove jargon ("credentials") 0.5sp ▼
Hypothesis
If we replace "credentials" with plain-language alternatives, then users feel less intimidated because "credentials" is IT language that increases perceived technical complexity at the highest-anxiety moment in the flow.

✓ If it wins
Plain language on the login subtitle reduces anxiety. Readability grade improvements at this step directly reduce hesitation.
✗ If it fails
Users don't process the subtitle carefully — they look at the form fields. Subtitle copy doesn't move the needle here.
#C10 Username field label — bank-agnostic copy 0.5sp ▼
Hypothesis

Variations to test
Username or ID Online ID or Username Dynamic per-bank label
✓ If it wins
Field label alignment with the user's bank terminology reduces credential entry errors. Small label mismatches cause outsized doubt at this step.
✗ If it fails
Users are familiar with their own bank's terminology — they enter their credential correctly regardless of our label.
#C11 Username helper text — reference frame 0.5sp ▼
Hypothesis

✓ If it wins
Reference-frame helper text reduces username entry hesitation without changing the label itself.
✗ If it fails
Users who know their credentials don't need the text; users who don't still need to look up their bank.
#C12 Password field label — terminology coverage 0.5sp ▼
Hypothesis

Variations to test
Password or PIN Password or Passcode Dynamic per-bank
✓ If it wins
Terminology coverage in field labels reduces credential friction. A label matching what the user's bank calls it eliminates "am I entering the right thing?" doubt.
✗ If it fails
Users know their own bank's login process — a label change doesn't help users who are simply unsure of their credentials.
#C13 Forgot password title — less abrupt, more orienting 0.5sp ▼
Hypothesis
If we rewrite the forgot password screen title to be more accurate ("Reset your password on your bank's site"), then users feel oriented rather than abandoned when they leave the flow, because the current title doesn't explain that they're temporarily leaving Trustly.
✓ If it wins
Accurate transition messaging reduces abandonment on the "forgot password" path. Users who understand they're going to their bank are more likely to return.
✗ If it fails
Users who tap "forgot password" have already decided to reset — the title doesn't change their path.
#C14 Login subtitle — remove warning tone 0.5sp ▼
Hypothesis
If we remove "please make sure" phrasing from the login subtitle, then users feel reassured rather than warned — "please make sure" reads like a warning about error risk rather than a confidence-building statement.
✓ If it wins
Removing warning-adjacent phrasing from trust copy increases perceived safety. The subtitle should feel like reassurance, not a caveat.
✗ If it fails
The phrasing is subtle enough that users don't interpret "please make sure" as a warning — the change is too minor to affect behaviour.
#C15 Sign in CTA — action-descriptive label 0.5sp ▼
Hypothesis
If we change "Agree & Sign in" to "Sign in" or "Sign in to my bank", then more users tap because the label describes the action rather than combining a legal commitment with a progression step.
Variations to test
Sign in Sign in to my bank Agree & Sign in (control)
✓ If it wins
Separating legal agreement from action CTAs increases tap rate. "Agree &" adds a contractual frame to what should feel like a simple action.
✗ If it fails
The "Agree" word isn't what's stopping users — they're hesitating about credentials, not about terms.
#C16 Invalid credentials error — generic but actionable 0.5sp ▼
Hypothesis
If we change the invalid credentials error to "Check your details and try again" (without specifying which field is wrong), then users retry rather than abandon — the message is calm and actionable without giving attackers field-level information.
Variations to test
Check your details and try again Check your info and try again Something doesn't match — try again
✓ If it wins
Generic-but-actionable error messages retain retrying users while maintaining security. "Check your details" is warm enough to feel helpful without helping bad actors.
✗ If it fails
Users need to know which field is wrong — a generic message increases frustration and consecutive failures.

OAuth

2 experiments
#C17 OAuth subtitle — specific action framing 0.5sp ▼
Hypothesis
If we replace "access your accounts" with "connect your account for this payment", then users feel less exposed because "access your accounts" implies broad, ongoing access rather than a single, scoped action.
✓ If it wins
Specificity in OAuth copy reduces perceived scope of permission. Users who understand they're connecting "for this payment" feel less like they're granting blanket access.
✗ If it fails
Users don't parse the difference — both feel like the same thing at this step.
#C18 OAuth subtitle — active, reduced copy 0.5sp ▼
Hypothesis
If we reduce the OAuth subtitle to an active, single-instruction sentence, then users scan and proceed faster because active voice and word reduction improve readability grade and lower cognitive load on the transition screen.
✓ If it wins
Shorter active copy on transition screens reduces time-to-tap. Users scan for the action signal, not full sentences.
✗ If it fails
The reduced copy removes context users need to feel comfortable with the redirect. Shorter isn't always better when trust is the primary job.

Dropoff Bottom Sheet

3 experiments
#C27 Dropoff title — name what's being left 0.5sp ▼
Hypothesis
If we change the dropoff title to name what the user is about to leave ("Cancel your deposit?"), then fewer users confirm the exit because making the consequence concrete activates loss aversion more effectively than a generic exit confirmation.
Variations to test
Cancel your deposit? Leave your payment? Sure you want to leave?
✓ If it wins
Naming the specific thing being abandoned is more effective at reversing exit intent than generic cancellation language.
✗ If it fails
Users who tap close have decided to leave — the title doesn't change that decision regardless of specificity.
#C28 Dropoff subtitle — remove "cancel" from body 0.5sp ▼
Hypothesis
If we remove the word "cancel" entirely from the dropoff body copy and focus instead on what's still possible ("You're almost there — your bank details are ready"), then fewer users exit because de-emphasising the exit while highlighting progress activates sunk-cost instinct.
✓ If it wins
Removing "cancel" from the body reduces salience of the exit option. "You're almost there" is more likely to reverse course than "cancel your session."
✗ If it fails
Users who opened the exit sheet know they want to leave — body copy reframing doesn't overcome actual exit intent.
#C29 Dropoff CTA — specific consequence label 0.5sp ▼
Hypothesis
If we make the exit CTA label specific about what will be cancelled ("Yes, cancel my deposit"), then cognitive friction of tapping a specific consequence reduces impulsive exits.
✓ If it wins
Specific consequence labels on exit CTAs reduce impulsive taps. "Yes, cancel my deposit" requires more deliberate commitment than "Leave" or "Cancel."
✗ If it fails
Users who tapped the exit affordance are already committed to leaving — the CTA label doesn't add enough friction to reverse the decision.

OLB / OAuth Redirect

2 experiments
#C32 Redirect subtitle — time expectation + return reassurance 0.5sp ▼
Hypothesis
If we rewrite the redirect subtitle to set a time expectation and make the return automatic ("This takes about 30 seconds — you'll be brought back here automatically"), then OAuth abandonment drops because users who know the return is automatic and time-bounded feel less uncertain about leaving the current screen.
✓ If it wins
Time expectation + automatic return copy reduces OAuth redirect abandonment. The two fears at this step — "how long?" and "will I get back?" — are both answered in one sentence.
✗ If it fails
Users who abandon at the redirect do so based on fundamental trust concerns — copy addressing timing doesn't overcome credential anxiety.
#C33 Redirect CTA — destination-named label 0.5sp ▼
Hypothesis
If we name the destination in the redirect CTA ("Sign in with Chase" instead of "Go to Chase"), then users feel oriented because naming the destination mirrors the SSO patterns users already know and is consistent with earlier OAuth language.
✓ If it wins
Destination-named CTAs on redirect screens reduce hesitation. Consistency with "Sign in with [bank]" framing creates a familiar, expected pattern.
✗ If it fails
The destination name is already visible in context — the CTA label change is too subtle to affect behaviour at this step.
